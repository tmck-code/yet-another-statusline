## Context

The wide top row's 5-hour segment is rendered by `Renderer.helper(five_hour, gap)` (`renderer.py:1567`). It computes the reset countdown from `resets_at - now` and prints `countdown = f'(-{h}:{m:02d})'` (`renderer.py:1588`), then appends `burndown_trend` (`renderer.py:1554`), which shows a `+X.X%` *deviation* glyph derived from `burndown_delta` (`render/metrics.py:8`, `used% - ideal%`). That deviation is a pace signal, **not** a rate — it cannot answer "when do I run out."

`helper` is reached via `build_wide` (`layout.py:423`) → `Renderer.model_right_section` (`renderer.py:567`) → `Renderer._rate_helpers` (`renderer.py:543`, calls `self.helper(rate_limits.five_hour, gap_5h)` at line 551). A second `_rate_helpers` call exists on the justify gap-widening path (`layout.py:545`). Both seams must thread any new argument.

The 5-hour bucket is `RateBucket` (`session.py:117`), read verbatim from the stdin `rate_limits` payload. It exposes only `used_percentage` (account-wide, shared across sessions) and `resets_at` — there are **no absolute token counts**, so a rate can only come from differencing `used_percentage` across renders.

The existing sampler to mirror is `TokenRate` (`tokens.py:159-201`): it read-modify-`write_text`s `statusline-token-rate.log` under `CLAUDE_DIR`, prunes rows older than `KEEP=300.0`s, and returns a first-vs-last delta over a config window (`token_window`, resolved lazily via `_token_window`). It is called from `record_tick` (`app.py:20`), which runs **only** in the wide layout; its result rides on `TickRecord` (`tokens.py:26`, `__slots__ = ('token_log', 'day_cost', 'tok_rate')`) through `build_wide` (`layout.py:401-403`).

The config idiom to mirror is `token_window` (`config.py`: field 351, `__slots__` 341, default-arg 372, `_resolve` 473-476 with `YAS_TOKEN_WINDOW`/`STATUSLINE_TOKEN_WINDOW` + `toml_src(tokens, 'token_window')`, wired into `cls(...)` at 539). Its README row is at line 77 and the `[tokens]` example at ~140.

## Goals / Non-Goals

**Goals:**
- Add a second, danger-coloured depletion countdown beside the reset countdown, gated so it only appears when a lockout is genuinely imminent.
- Model the burn rate **instantaneously** (`Δused% / Δt`) from a short lookback, distinct from the existing deviation.
- Keep all disk I/O in the app/tokens layer; the renderer stays pure and takes a precomputed rate.
- Preserve today's `(-{h}:{m:02d})` rendering byte-identically whenever the depletion segment is suppressed (medium/narrow layouts, no samples, not burning, or depletion after reset).

**Non-Goals:**
- No window-average or EMA rate model — instantaneous first-vs-last only (Decision 1).
- No per-session keying of the 5-hour series — it is account-wide, so the series is global (Decision 2).
- No reuse of `burndown_delta` / `burndown_trend` as a rate; the deviation glyph is untouched.
- No change to the 7-day segment, the compact/medium 5-hour rendering, or narrow layouts.
- No new border divider/elbow — the depletion segment lives inside the existing 5-hour cell text.

## Decisions

**1. Instantaneous burn-rate model.** Sample the 5-hour `used_percentage` over time; `rate = (used_last - used_first) / ((t_last - t_first) / 60)` in %/min, taken first-vs-last across the lookback. Not a window average, not the deviation. If fewer than 2 samples remain in-window, or the span `< DT_FLOOR`, the rate is unavailable (return `0`/`None`).

**2. `FiveHourRate` sampler (`tokens.py`, mirroring `TokenRate`).** Each wide render, append `(ts, resets_at, used_pct)` to a new log `CLAUDE_DIR / 'statusline-5h-rate.log'`. The series is **global** — rows are NOT keyed by `session_id`, because `RateBucket.used_percentage` is account-wide and any session observes the same 5-hour bucket. Rollover handling: filter retained rows to those whose `resets_at` equals the *current* `resets_at`, so when the window rolls over (a new `resets_at`) the prior series is discarded and a fresh one starts — this also drops stale samples from an expired window. Pruning: drop rows older than a `KEEP` retention; retention MUST be `>=` the max lookback so needed rows survive (e.g. `KEEP = max(600.0, window)` or a fixed value comfortably above the 300 s default). `update(resets_at, used_pct) -> float | None` returns the %/min rate (or `None`/`0` when <2 in-window samples or span `< DT_FLOOR`). Guard: `resets_at == 0` (no window) short-circuits to no rate, matching how `TokenRate.update` bails on an empty `session_id`.

**3. Config knob `five_hour_rate_window`.** Mirror `token_window` exactly: `Config` field + `__slots__` entry + default-arg (`DEFAULT_FIVE_HOUR_RATE_WINDOW`) + `_resolve('five_hour_rate_window', _env_sources(env, 'YAS_FIVE_HOUR_RATE_WINDOW', 'STATUSLINE_FIVE_HOUR_RATE_WINDOW') + toml_src(tokens, 'five_hour_rate_window'), _parse_pos_float, DEFAULT_FIVE_HOUR_RATE_WINDOW, ...)` + wiring into `cls(...)` + a README knob row and `[tokens]` example line. Default = `300` (seconds; the ~5-min lookback). `FiveHourRate` resolves it lazily via a `functools.cache`d `_five_hour_rate_window()` helper mirroring `_token_window` (`tokens.py:153`), with a class `WINDOW: float | None = None` override for tests. `DT_FLOOR` (minimum Δt span, e.g. `60.0` s) is a plain constant in `constants.py` — it is a correctness guard, not a user knob.

**4. Depletion math — pure helper in `render/metrics.py`.** Add beside `burndown_delta`, same style/signature discipline (unit-testable, no I/O):
`deplete_minutes(used_pct: float, rate_per_min: float | None) -> float | None` returning `(100 - used_pct) / rate_per_min` when `rate_per_min` is a positive float, else `None`. This is the single place the depletion span is computed; the renderer only formats it.

**5. Wiring — I/O in the app layer, rate threaded to the renderer.** `record_tick` (`app.py:20`) calls `FiveHourRate.update(session.rate_limits.five_hour.resets_at, session.rate_limits.five_hour.used_percentage)` and stores the result on a new `TickRecord` slot `five_h_rate` (add `'five_h_rate'` to `__slots__` and the `__init__` signature in `tokens.py:26-32`, defaulting to `None`). `build_wide` reads `tick.five_h_rate` (`layout.py:401` block) and passes it into `r.model_right_section(..., five_h_rate=tick.five_h_rate)`; `model_right_section` forwards it to `_rate_helpers(rate_limits, five_h_rate=...)`, which forwards it to `helper(five_hour, gap_5h, five_h_rate=...)`. The gap-widen re-call at `layout.py:545` passes `tick.five_h_rate` too. **`helper`'s `five_h_rate` parameter defaults to `None`** so the medium/narrow builders (which have no `TickRecord` and call `model_right_section` / `helper` without it) simply render the reset countdown alone — no disk read, no rate.

**6. Show condition (in `helper`, at the countdown construction ~`renderer.py:1588`).** After computing `remain_min` (= `total_s / 60`) and `deplete_min = deplete_minutes(used_pct, five_h_rate)`, show the depletion part **only** when ALL hold: (a) `five_h_rate` is a positive float (used% increasing → `deplete_min` is a positive float, so this collapses to `deplete_min is not None`), AND (b) `deplete_min < remain_min` (would hit 100% before the window resets), AND (c) the rate came from ≥2 valid samples spanning ≥ `DT_FLOOR` — which `FiveHourRate.update` already enforces by returning `None` otherwise, so `five_h_rate is None` covers it. When shown, format the depletion span with the **same `divmod` floor logic** as the reset countdown: `dh, drem = divmod(int(deplete_min * 60), 3600); dm = drem // 60`, yielding `(-{h}:{m:02d}/-{dh}:{dm:02d})`. Otherwise render exactly `(-{h}:{m:02d})` as today.

**7. Format & colour.** One parenthesis group, no extra spaces: `(-{h}:{m:02d}/-{dh}:{dm:02d})`. The reset portion and the opening/closing parens stay in `self.COMMIT` (today's colour); only the `/-{dh}:{dm:02d}` depletion segment is wrapped in a WARN colour — reuse the near-100% fill colour (e.g. `self.fill_colour(100.0)` / the red end of the fill gradient) so it reads as danger — with an explicit reset back to `self.COMMIT` before the closing paren. Concretely: `f'{self.COMMIT}(-{h}:{m:02d}{warn}/-{dh}:{dm:02d}{self.COMMIT})'`. The subsequent `{self.R}{sp}{pct_clr}{pct_str}%...` tail is unchanged.

**8. Tests.** `test/test_burndown.py` for `deplete_minutes` (positive rate → span; `None`/zero/negative rate → `None`; boundary `used_pct=100`). New `FiveHourRate` unit tests (in `test_burndown.py` or `test_tokens_cost.py`, using the `tmp_home` fixture that patches `CLAUDE_DIR`): global keying (two different `session_id`s contribute to one series), `resets_at` rollover discards the stale series, <2 samples → `None`, Δt-floor guard → `None`. `test_config.py` for the new knob (default 300, env override via both `YAS_*` and legacy alias, toml `[tokens].five_hour_rate_window`, invalid value falls back to default). `test/test_helper.py`: update the format-pinning assertions (they pin `(-1:00)`, `(-2:00)`, `(-0:45)` etc. via `helper` with no `five_h_rate`) so the no-depletion path still matches byte-for-byte, and add cases for the combined `(-0:11/-0:04)` form with a supplied `five_h_rate` (monkeypatching `datetime.now`/`time.time` as the existing tests do).

## Risks / Trade-offs

- **Header width grows (~7 → ~13 visible chars).** `(-0:11)` → `(-0:11/-0:04)` widens the 5-hour cell in the wide layout, which is exactly the class of change the `make demo/img` visual/column-width gate exists to catch. All width math for the cell MUST go through `_visible_width` (`render/text.py`), never `len()`, and the WARN-colour ANSI wrapping MUST NOT leak into any column count. The `verifier` gate runs `make test` **plus** `make demo/img` for this reason; expect elbow/`┬`/`┴` alignment shifts in the 5-hour segment to need eyeballing.
- **Instantaneous rate is noisy.** First-vs-last over ~5 min can swing on a single burst. Accepted: the gate `deplete_min < remain_min` plus the `DT_FLOOR` span floor suppress the segment except when depletion is genuinely sooner than reset, and it silently vanishes when the burst stops (rate ≤ 0 → no rate).
- **Global-series contention.** Multiple concurrent sessions all write the one global log every wide render. This matches the account-wide semantics of `used_percentage` (all sessions observe the same bucket) and mirrors `TokenRate`'s single-file pattern; the read-modify-write is small and last-writer-wins is acceptable for a monotonic percentage series.
- **Retention vs. lookback coupling.** If `KEEP < window` a needed first sample could be pruned before it is used, silently zeroing the rate. Mitigation: `KEEP` is derived as `>=` the resolved window (Decision 2), and a test pins that an in-window sample is never pruned.
- **`used_percentage` is coarse (2 dp, rounded in `RateBucket.from_dict`).** Over a short span the delta may be `0.0` even while burning, yielding no rate. Accepted — a zero delta correctly reads as "not measurably burning right now," and the segment reappears once the percentage ticks up.
